# HIPAA ePHI Protection

Demonstrate HIPAA Security Rule compliance monitoring for workloads that handle electronic Protected Health Information (ePHI).

## What This Demonstrates

- ArgusWatcher file integrity monitoring for HIPAA 164.312(c)(1) -- Data Integrity
- JanusGuard access auditing for HIPAA 164.312(b) -- Audit Controls
- Detection of unauthorized access to database files and credential stores
- Compliance tagging with HIPAA requirement IDs

## HIPAA Context

The HIPAA Security Rule requires three categories of safeguards for systems handling ePHI:

| Safeguard | Requirement | Panoptes Coverage |
|-----------|-------------|-------------------|
| **Audit Controls** | 164.312(b) -- Record and examine activity | JanusGuard access auditing on data directories |
| **Data Integrity** | 164.312(c)(1) -- Protect from improper alteration | ArgusWatcher FIM on config, logs, and data files |
| **Authentication** | 164.312(d) -- Verify identity of accessors | ArgusWatcher on /etc/pam.d, /etc/shadow |

Panoptes provides the technical controls that satisfy these requirements. Each event is tagged with the specific HIPAA requirement ID for audit trail purposes.

## Step-by-Step Walkthrough

### Step 1: Install Panoptes with HIPAA Compliance

```bash
helm install panoptes oci://ghcr.io/como-technologies/charts/panoptes \
  -n panoptes-system --create-namespace \
  --set compliance.hipaa.enabled=true
```

This automatically creates HIPAA-specific ArgusWatcher and JanusGuard resources that target pods labeled `hipaa/scope: ephi`.

### Step 2: Deploy an ePHI Workload

```bash
kubectl apply -f ephi-workload.yaml
kubectl wait --for=condition=available deploy/ephi-database --timeout=120s
```

This deploys a PostgreSQL-based database labeled `hipaa/scope: ephi`, simulating a system that stores patient health records.

### Step 3: Verify HIPAA Monitoring

```bash
kubectl get aw,jg -l compliance=hipaa
```

Expected output:
```
NAME                                    AGE
arguswatcher.argus.como-technologies.io/hipaa-fim   30s

NAME                                   AGE
janusguard.janus.como-technologies.io/hipaa-access   30s
```

### Step 4: Simulate ePHI Access and Integrity Violations

```bash
POD=$(kubectl get pods -l app=ephi-database -o jsonpath='{.items[0].metadata.name}')

# Simulate unauthorized credential access (164.312(d))
kubectl exec "$POD" -- cat /etc/shadow

# Simulate config tampering (164.312(c)(1))
kubectl exec "$POD" -- bash -c "echo 'host all all 0.0.0.0/0 trust' >> /var/lib/postgresql/data/pg_hba.conf"

# Simulate log deletion (164.312(c)(1))
kubectl exec "$POD" -- bash -c "rm -f /var/log/postgresql/postgresql.log"

# Simulate authentication config change (164.312(d))
kubectl exec "$POD" -- bash -c "echo 'auth sufficient pam_permit.so' > /etc/pam.d/common-auth"
```

### Step 5: View HIPAA Audit Events

```bash
# ArgusWatcher events (file integrity)
kubectl logs -n panoptes-system -l app=argusd --tail=20

# JanusGuard events (access audit)
kubectl logs -n panoptes-system -l app=janusd --tail=20
```

Expected log events:
```
{"event":"modify","path":"/var/lib/postgresql/data/pg_hba.conf","tags":{"requirement":"164.312(c)(1)","severity":"high"}}
{"event":"access","path":"/etc/shadow","tags":{"requirement":"164.312(d)","severity":"critical"}}
{"event":"delete","path":"/var/log/postgresql/postgresql.log","tags":{"requirement":"164.312(c)(1)","severity":"high"}}
{"event":"modify","path":"/etc/pam.d/common-auth","tags":{"requirement":"164.312(d)","severity":"critical"}}
```

## Auditor-Friendly Output

Every event generated by Panoptes includes:

- **Timestamp**: When the event occurred (UTC)
- **Event type**: What happened (modify, delete, access, open)
- **Path**: Which file was affected
- **Pod**: Which pod generated the event
- **Tags**: HIPAA requirement ID, severity, category

This data can be exported to your SIEM or log aggregator for audit trail retention (HIPAA requires 6 years of audit log retention).
